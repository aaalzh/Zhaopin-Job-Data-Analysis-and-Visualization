# 智联招聘爬虫数据分析系统需求文档

## 1. 项目概述

### 1.1 项目名称

智联招聘岗位数据爬虫与大数据分析系统

### 1.2 项目背景

随着企业招聘信息和岗位技能要求不断变化，招聘网站中的岗位数据能够反映不同城市、不同行业、不同岗位方向的人才需求情况。本项目以“智联招聘”网站公开岗位信息为数据来源，围绕岗位名称、薪资、城市、学历、经验、公司类型、公司规模、福利标签、岗位描述等字段进行采集、清洗、存储、分析和展示。

项目结合 Python 爬虫、Redis 任务缓存、Hadoop HDFS 分布式存储、Hive 数据仓库、MySQL 结果存储和 ECharts 数据可视化等技术，构建一套面向实训课程的大数据采集与分析系统。

### 1.3 项目目标

1. 实现智联招聘岗位数据的自动化采集，支持按关键词、城市、页数和时间批次抓取数据。
2. 对原始岗位数据进行清洗、去重、标准化和结构化处理，形成可分析的数据集。
3. 使用 Hadoop HDFS 存储原始数据和清洗数据，使用 Hive 建立岗位数据仓库。
4. 基于 Hive 完成岗位数量、薪资水平、城市分布、学历要求、经验要求、企业类型、技能关键词等统计分析。
5. 将核心分析结果写入 MySQL，供 Python 查询脚本、报表程序或 ECharts 可视化页面查询展示。
6. 使用 Redis 管理爬虫任务、去重标记、失败重试队列和热点查询缓存。
7. 输出项目文档、数据字典、分析结果和 ECharts 可视化图表，满足课程实训展示与答辩要求。

## 4. 业务范围

### 4.1 数据采集范围

系统采集智联招聘网站公开展示的岗位搜索结果数据，主要支持以下采集维度：

| 维度 | 示例 |
| --- | --- |
| 岗位关键词 | 大数据、Python、数据分析、人工智能、运维 |
| 城市 | 重庆、成都、北京、上海、深圳、杭州 |
| 页码范围 | 第 1 页至第 N 页 |
| 采集批次 | 按采集日期和任务编号区分 |
| 岗位状态 | 正常展示的招聘岗位 |

### 4.2 数据分析范围

系统需要完成以下分析主题：

1. 不同城市岗位数量分布分析。
2. 不同岗位关键词的招聘需求对比。
3. 岗位薪资区间分布分析。
4. 平均薪资、最低薪资、最高薪资统计。
5. 学历要求分布分析。
6. 工作经验要求分布分析。
7. 公司规模、公司性质、行业类型分析。
8. 岗位福利标签高频词分析。
9. 岗位描述中的技能关键词分析。
10. 按采集批次对岗位数量变化进行对比分析。

## 5. 总体架构

### 5.1 架构说明

系统采用“采集层、缓存层、存储层、数仓层、结果输出层、展示层”的分层架构。

```mermaid
flowchart LR
    A["智联招聘公开岗位数据"] --> B["Python 爬虫采集"]
    B --> C["Redis 队列/去重/缓存"]
    B --> D["本地 CSV/JSON 原始文件"]
    D --> E["HDFS 原始数据目录"]
    E --> F["Hive ODS 原始层"]
    F --> G["Hive DWD 清洗明细层"]
    G --> H["Hive DWS/ADS 统计分析层"]
    H --> I["MySQL 分析结果库"]
    I --> J["Python 查询/报表脚本"]
    J --> K["ECharts 可视化页面/报表展示"]
```

### 5.2 分层职责

| 层级 | 职责 |
| --- | --- |
| 采集层 | 使用 Python 请求岗位数据，解析 JSON 或页面数据，生成原始数据文件 |
| 缓存层 | 使用 Redis 存放任务队列、已采集岗位 ID、失败任务、限速控制信息 |
| 存储层 | 使用 HDFS 保存原始数据、清洗后数据和分析中间数据 |
| 数仓层 | 使用 Hive 建立 ODS、DWD、DWS、ADS 分层数据表 |
| 结果输出层 | 使用 Python 读取 MySQL 中的分析结果，生成报表数据、图表数据或轻量查询接口 |
| 展示层 | 使用 ECharts 展示岗位分布、薪资分布、学历经验要求、技能词频等图表 |

## 6. 功能需求

### 6.1 爬虫任务管理

#### 6.1.1 任务配置

系统应支持配置以下爬虫参数：

| 参数 | 说明 |
| --- | --- |
| keyword | 岗位关键词 |
| city | 目标城市 |
| page_start | 起始页码 |
| page_end | 结束页码 |
| page_size | 每页数量 |
| crawl_interval | 请求间隔 |
| max_retry | 最大重试次数 |
| output_format | 输出格式，支持 CSV、JSON、XLSX |

#### 6.1.2 任务执行

1. 系统应支持按任务配置自动生成采集请求。
2. 系统应支持多关键词、多城市批量采集。
3. 系统应记录每个任务的开始时间、结束时间、采集页数、成功条数、失败条数。
4. 系统应支持失败任务重试。
5. 系统应支持根据岗位 ID 或岗位链接进行去重。
6. 系统应限制请求频率，避免对目标网站造成过高访问压力。

#### 6.1.3 异常处理

1. 当页面出现安全验证、接口失效、请求超时、返回空数据时，系统应记录异常日志。
2. 当 HTML 页面无法解析时，系统应优先尝试 JSON 接口采集。
3. 当接口字段发生变化时，系统应保留原始 JSON 数据，便于后续重新解析。
4. 当连续失败次数超过阈值时，系统应暂停该任务并标记为异常。

### 6.2 数据采集

#### 6.2.1 采集字段

系统至少需要采集以下字段：

| 字段名 | 中文含义 | 示例 |
| --- | --- | --- |
| job_id | 岗位 ID | CC123456 |
| job_name | 岗位名称 | 大数据开发工程师 |
| company_name | 公司名称 | 某科技有限公司 |
| city | 工作城市 | 重庆 |
| district | 工作区域 | 渝北区 |
| salary | 原始薪资 | 8千-1.2万 |
| salary_min | 最低薪资 | 8000 |
| salary_max | 最高薪资 | 12000 |
| salary_avg | 平均薪资 | 10000 |
| education | 学历要求 | 本科 |
| experience | 经验要求 | 3-5年 |
| company_type | 公司性质 | 民营 |
| company_size | 公司规模 | 100-299人 |
| industry | 行业 | 互联网/电子商务 |
| welfare | 福利标签 | 五险一金、周末双休 |
| job_desc | 岗位描述 | 负责数据平台建设 |
| publish_time | 发布时间 | 2026-06-15 |
| crawl_time | 采集时间 | 2026-06-15 10:00:00 |
| source_url | 来源链接 | https://www.zhaopin.com/ |

#### 6.2.2 原始数据保留

系统应保留完整原始响应数据，包括：

1. 原始 JSON 数据。
2. 展平后的原始字段表。
3. 每页采集日志。
4. 请求参数和返回状态。

### 6.3 数据清洗

系统应对采集数据进行清洗，清洗规则如下：

| 清洗项 | 处理方式 |
| --- | --- |
| 重复岗位 | 按 job_id、company_name、job_name、city 组合去重 |
| 空值字段 | 核心字段为空的数据标记为异常，非核心字段填充为“未知” |
| 薪资字段 | 将“8千-1.2万”“15-20K”等文本转换为数值 |
| 城市字段 | 统一城市名称，去除多余区域描述 |
| 学历字段 | 统一为不限、中专、大专、本科、硕士、博士 |
| 经验字段 | 统一为不限、应届、1年以内、1-3年、3-5年、5-10年、10年以上 |
| 福利标签 | 拆分为数组或词频字段 |
| 岗位描述 | 去除 HTML 标签、特殊字符和多余空格 |

### 6.4 HDFS 存储

HDFS 目录规划如下：

```text
/zhaopin/
  raw/
    crawl_date=2026-06-15/
  clean/
    crawl_date=2026-06-15/
  hive/
  result/
  logs/
```

存储要求：

1. 原始数据按采集日期分区保存。
2. 清洗数据按采集日期、城市、关键词分区保存。
3. 文件格式初期支持 CSV/JSON，后续可扩展为 Parquet。
4. HDFS 文件命名应包含关键词、城市、采集日期和任务编号。

#### 6.5.1 数仓分层

| 层级 | 表名示例 | 说明 |
| --- | --- | --- |
| ODS | ods_zhaopin_job_raw | 原始采集数据 |
| DWD | dwd_zhaopin_job_clean | 清洗后的岗位明细数据 |
| DWS | dws_zhaopin_city_salary | 城市薪资统计汇总 |
| DWS | dws_zhaopin_education_exp | 学历和经验统计汇总 |
| ADS | ads_zhaopin_dashboard | 可视化看板结果表 |

#### 6.5.2 ODS 原始表

```sql
CREATE TABLE IF NOT EXISTS ods_zhaopin_job_raw (
    raw_json STRING,
    source_url STRING,
    keyword STRING,
    city STRING,
    page_index INT,
    crawl_time STRING
)
PARTITIONED BY (crawl_date STRING)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY '\t'
STORED AS TEXTFILE;
```

#### 6.5.3 DWD 清洗明细表

```sql
CREATE TABLE IF NOT EXISTS dwd_zhaopin_job_clean (
    job_id STRING,
    job_name STRING,
    company_name STRING,
    city STRING,
    district STRING,
    salary STRING,
    salary_min DOUBLE,
    salary_max DOUBLE,
    salary_avg DOUBLE,
    education STRING,
    experience STRING,
    company_type STRING,
    company_size STRING,
    industry STRING,
    welfare STRING,
    job_desc STRING,
    publish_time STRING,
    crawl_time STRING,
    source_url STRING

PARTITIONED BY (crawl_date STRING, keyword STRING)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY '\t'

```

#### 6.5.4 统计分析需求

Hive 需要支持以下统计 SQL：

1. 按城市统计岗位数量。
2. 按城市统计平均薪资、最高薪资、最低薪资。
3. 按学历统计岗位数量占比。
4. 按经验要求统计岗位数量占比。
5. 按公司规模统计岗位数量。
6. 按公司类型统计岗位数量。
7. 按关键词统计招聘需求热度。
8. 按福利标签统计高频福利。
9. 按岗位描述统计技能关键词词频。
10. 按采集日期统计岗位数量变化趋势。

## 7. 数据流程

### 7.1 离线处理流程

1. 管理员配置采集任务。
2. Python 爬虫从 Redis 队列读取任务。
3. 爬虫请求智联招聘公开岗位数据。
4. 原始 JSON、清洗 CSV、采集日志写入本地文件。
5. 数据上传至 HDFS。
6. Hive 加载 ODS 原始表。
7. Hive SQL 清洗生成 DWD 明细表。
8. Hive SQL 聚合生成 DWS/ADS 统计表。
9. 分析结果同步写入 MySQL。
10. Python 查询脚本读取 MySQL 并生成报表数据或图表数据。
11. ECharts 前端页面展示统计图表和报表。

### 7.2 数据流转示例

```text
智联招聘岗位数据
  -> Python 爬虫
  -> Redis 去重与任务管理
  -> 本地 raw/clean/meta 文件
  -> HDFS
  -> Hive ODS
  -> Hive DWD
  -> Hive DWS/ADS
  -> MySQL
  -> Python 查询/可视化数据输出
  -> ECharts 可视化页面
```

## 8. 非功能需求

### 8.1 性能需求

1. 单次任务应支持采集至少 100 条岗位数据。
2. 系统应支持按多个城市和多个关键词批量采集。
3. Hive 查询常用统计结果的执行时间应控制在可接受范围内。
4. Python 查询脚本或 ECharts 可视化页面读取 MySQL 统计结果时，普通查询响应时间应小于 3 秒。

### 8.2 稳定性需求

1. 爬虫请求失败后应自动重试。
2. Redis 任务队列应避免重复采集。
3. HDFS 和 Hive 路径应按日期分区，避免数据混乱。
4. 每次采集必须生成日志，便于定位异常。

### 8.3 安全与合规需求

1. 仅采集网站公开展示的岗位信息，不采集个人隐私数据。
2. 控制采集频率，不对目标网站造成高频访问压力。
3. 不绕过登录、付费、验证码或权限控制。
4. 数据仅用于课程实训、学习分析和本地展示。
5. 报告中应注明数据来源、采集时间和使用范围。

### 8.4 可维护性需求

1. 爬虫请求参数、城市编码、关键词列表应配置化。
2. 数据清洗规则应独立封装，便于修改。
3. Hive SQL 脚本应按 ODS、DWD、DWS、ADS 分目录管理。
4. Python 查询脚本和 ECharts 可视化数据输出应保持统一字段格式。
5. ECharts 图表配置应按图表类型独立维护，便于调整标题、坐标轴、图例和数据映射。
5. 日志应包含任务 ID、关键词、城市、页码、状态、错误信息。

## 11. 关键业务规则

1. 同一岗位在同一采集批次中只能保留一条有效记录。
2. 薪资无法解析时，保留原始薪资文本，数值薪资字段置空。
3. 学历和经验字段必须统一分类后再进入统计表。
4. 城市为空或岗位名称为空的数据不得进入 DWD 明细表。
5. 每个采集批次必须保留原始数据，便于后续追溯。
6. Hive 分区字段必须包含 crawl_date，便于按日期管理数据。
7. MySQL 只保存统计结果和任务信息，不保存大量原始岗位明细。

## 14. 风险与解决方案

| 风险 | 影响 | 解决方案 |
| --- | --- | --- |
| 网站页面结构变化 | 爬虫解析失败 | 保留原始 JSON，解析逻辑模块化，及时调整字段映射 |
| 出现安全验证 | 无法继续采集 | 降低请求频率，暂停任务，优先使用公开接口或人工导出样例数据 |
| 字段缺失或格式不统一 | 影响统计准确性 | 建立清洗规则和异常数据表 |
| Hadoop/Hive 环境配置复杂 | 影响项目进度 | 先完成本地 CSV 分析，再逐步接入 HDFS 和 Hive |
| 数据量过小 | 分析结果说服力不足 | 增加关键词、城市和采集批次 |
| MySQL 与 Hive 数据同步失败 | 页面无结果展示 | 增加同步日志和失败重跑脚本 |
